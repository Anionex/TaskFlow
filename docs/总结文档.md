## 1. 需求文档

> **本章完成人：角色1（产品经理 + 后端开发）**

> 说明：本节为 V2.0「智能化升级」后的需求文档。V1.0 已实现一套完整的传统任务管理系统（任务 CRUD、批量操作、回收站、循环模板、打卡、统计、导入导出、多主题）；V2.0 在此基础上引入大模型能力，将产品重新定位为「智能化 GTD 系统」，并将架构升级为中央数据库 + 部署式后端，交付形态升级为「原生桌面应用 + 网页托管版」双形态。

### 1.1 项目背景

传统待办软件的核心问题是**使用成本太高**：创建任务时要手动填写标题、分类、时间、优先级；面对大目标时不知道如何拆解；任务一多就难以分类、检索，也难以判断"今天该做什么"。

TaskFlow V2.0 将自身重新定位为**智能化 GTD 系统**，核心理念是"**用户少操作，系统多理解**"：利用大模型的自然语言理解能力，把用户随口说出的想法自动整理成**结构化、可执行、可推进**的任务系统。用户只需用自然语言表达，系统负责解析、分类、拆解、检索与推进。


### 1.2 目标用户

- 学生：管理课程作业、考试复习、课程设计、答辩等

- 上班族：管理日常工作任务、项目进度等

- 个人用户：管理生活琐事、个人目标等

- 尤其适合任务多、但不愿在"录入与整理"上花费精力的用户

### 1.3 功能需求

本期功能分为「智能能力（V2.0 新增）」与「基础任务管理（V1.0 沿用）」两部分。

#### 1.3.1 智能任务录入（核心）

- 自然语言解析：用户输入如"明天下午把数据库课设报告初稿写完，放到学习分类，优先级高一点"，系统自动解析出任务标题、分类、重要程度（星级）、开始/截止时间等字段

- 解析确认与编辑：解析结果以**可编辑草稿**形式呈现，用户一键微调后确认入库，系统**不静默写入**

- 自动分类：用户未指定分类时，系统自动判断任务属于 学习/工作/生活/家庭/其他；用户显式指定分类时以用户为准

- 批量捕获（Brain dump）：用户一次性输入一段包含多件事的文字，系统拆成多条任务草稿并一并确认

- 下一步行动改写：识别"含糊、不可执行"的任务（如"搞定毕业的事"），建议改写为具体可执行的下一步动作

#### 1.3.2 大任务拆解

- 大目标识别：当用户输入"完成课程设计""准备期末考试""写毕业论文"这类大目标时，系统识别其并非单一任务，并主动询问是否需要拆解

- 拆解生成：用户确认后，系统生成多个可执行的子任务，并给出建议执行顺序

- 任务组与一级子任务：拆解结果作为一个"任务组"统一管理，父任务作为容器并展示完成进度（如 3/5）；**仅支持一级子任务**，不做无限层级，保持系统轻量

#### 1.3.3 智能检索

- 语义检索：支持自然语言查询，如"找出这周要完成的学习任务""有哪些和答辩有关的事情""快到期但还没做的任务"

- 系统应理解时间、状态、分类、优先级与语义相关性，而非仅做关键词匹配

#### 1.3.4 智能推进（人文关怀）

- 早间任务推荐：综合截止时间、重要程度、过期任务、子任务建议顺序、昨日未完成任务等，推荐今天最值得做的 3-5 件事，并说明推荐理由

- 晚间今日总结：基于当天完成、未完成、新增、延期的任务，生成今日回顾与推进建议（基于任务数据，非日记）

- 语气设置：推荐与总结的语气允许用户设置，如"温暖鼓励型""冷静督促型""简短效率型"

#### 1.3.5 基础任务管理（V1.0 沿用）

- 创建任务：标题、描述、分类、星级、开始/截止日期

- 编辑、删除（移至回收站）、完成/未完成切换

- 按分类筛选、按创建时间/截止日期/星级排序、关键词搜索、分页

#### 1.3.6 批量操作

- 批量选择、批量删除、一键清空待办/已完成/已过期列表

#### 1.3.7 任务统计

- 任务概览（总数/待办/完成）、饼图（完成/待做/过期占比）、柱状图（近 12 个月完成量）

#### 1.3.8 回收站

- 任务恢复、永久删除、清空回收站

#### 1.3.9 循环任务模板（前端更名为「习惯」）

- 模板（前端界面称"习惯"）的创建/编辑/删除，按日/周/月自动生成任务，可配置生成与截止时间；同一习惯每天至多生成一次（幂等）

#### 1.3.10 每日打卡

- 每日签到、当前连续与最长连续天数统计、签到激励语

#### 1.3.11 数据管理

- 任务数据的导入与导出

#### 1.3.12 用户管理

- 手机号注册（11 位校验）、手机号+密码登录、会话管理、修改密码

#### 1.3.13 主题切换

- 普通模式、护眼模式、夜间模式

#### 1.3.14 智能助理（Agent 对话模式）

- 对话式任务管家：用户用自然语言对话完成任务的增删改查，如"这周学习类还剩哪些没做""把明天的会议改到后天下午""帮我加一个周五交报告的任务"

- 多轮工具调用：助理通过工具（list_tasks / get_stats / create_task / update_task / delete_task）读取真实任务数据后再作答，不凭空编造 id；一次只调一个工具，拿到结果再决定下一步

- 写操作确认闸：新增/修改/删除不静默执行，先作为"拟改动"卡片展示，用户点确认后才真正落库；用户可拒绝并要求助理调整，助理不会原样重复同一改动

- 流式输出：助理的思考过程、工具调用轨迹与最终答复以 SSE 流式返回，前端实时呈现（思考/工具可折叠），最终答复支持 Markdown 渲染

- 无状态设计：对话历史由前端持有并随每次请求回传，后端不保存会话，便于水平扩展

#### 1.3.15 范围边界（本期暂不纳入）

为保持 MVP 轻量、避免演变为复杂项目管理工具，以下内容明确推迟或排除：

- **任务依赖**（前置/后继约束）：本期不做依赖引擎，仅以"建议顺序"展示；后续如确有需求再评估

- **任务清理**（自动识别长期不动/含糊/重复任务并建议归档合并）、**超载检查**（耗时估算与现实排期）：列为下一期能力

- 日记、情绪分析、长期行为画像、复杂多层级项目管理、团队协作：不在本产品主线内

### 1.4 非功能需求

#### 1.4.1 性能需求

- 普通（非大模型）API 响应时间不超过 500 毫秒

- 大模型相关操作（解析、拆解、检索、推荐、总结）属于在线推理，响应允许达到数秒级，但须提供明确的加载状态提示

- 支持同时处理多个用户请求

#### 1.4.2 智能能力可靠性需求

- 大模型输出必须经用户确认后才落库，杜绝静默写入

- 大模型不可用或超时时，须优雅降级到手动表单，核心任务管理功能不受影响

- 结构化解析采用大模型的工具调用（tool use）/ JSON Schema 约束输出，提升解析稳定性

#### 1.4.3 安全与隐私需求

- 密码采用 Argon2 加盐哈希存储（替代 V1.0 的 SHA-256 裸哈希）

- 会话持久化存储于中央数据库（Supabase / PostgreSQL），服务重启后不丢失

- 任务内容会被发送至大模型服务进行处理，须在产品中明确告知用户

- 大模型 API 密钥由用户自行提供，支持两种保管方式：① 仅客户端本地保管、随请求头临时传递、后端用完即弃；② 保存到账户（持久化于 users 表）以实现跨设备同步。后端解析配置的优先级为「请求头 > 账户设置 > 服务端默认」，用户可随时清空

- 数据库访问统一使用参数化查询防止注入；前端输出转义防止 XSS

#### 1.4.4 可用性需求

- 以**原生桌面应用 + 网页托管版**双形态交付；桌面端拥有独立窗口与应用图标，而非调起系统浏览器；两端共用同一前端代码与后端服务

- 界面简洁直观，支持普通/护眼/夜间多主题

- 提供友好的错误与加载提示

#### 1.4.5 可维护性需求

- 前后端分层：前端 React（网页版与 Tauri 桌面版共用同一套代码）+ 部署式后端 Rust/Axum + 中央数据库 Supabase（PostgreSQL）

- 数据访问层与大模型调用层模块化，便于替换与测试

- 使用版本控制工具管理代码，保留必要注释与文档

## 2. 设计文档

> **本章完成人：角色1（系统架构、数据模型）、角色4（模块/界面设计文档编写）**

### 2.1 系统架构

系统采用「双客户端 + 部署式后端 + 中央数据库」的三层架构：网页版与桌面版共用同一套 React 前端，统一通过 HTTPS 调用部署的 Axum 后端；后端是唯一的业务与智能逻辑入口，向下连接 Supabase（PostgreSQL）作为唯一数据源，向外通过 OpenAI 兼容接口调用大模型。

```
┌──────────────────────────┐        ┌──────────────────────────┐
│   网页版前端 (React)       │        │  桌面版 (Tauri + React)    │
│   浏览器中运行             │        │  原生窗口，复用同一前端     │
└─────────────┬────────────┘        └────────────┬─────────────┘
              └────────────── HTTPS / JSON ───────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│               部署式后端服务 (Rust + Axum)                    │
│   认证/会话 · 任务/子任务 · 批量/回收站 · 模板/打卡           │
│   统计 · 导入导出 · 智能服务(解析/拆解/检索/推荐/总结)        │
└──────────────┬───────────────────────────┬──────────────────┘
   SQL (sqlx)  │                            │  OpenAI 兼容 API
               ▼                            ▼
┌──────────────────────────┐    ┌──────────────────────────────┐
│  Supabase (PostgreSQL)    │    │  大模型服务 (OpenAI 兼容)      │
│  users/sessions/tasks/... │    │  默认 deepseek-v4-flash        │
└──────────────────────────┘    └──────────────────────────────┘
```

### 2.2 技术选型

#### 2.2.1 后端技术

- **编程语言：** Rust

- **Web框架：** Axum 0.8

- **异步运行时：** Tokio

- **序列化：** Serde + serde_json

- **数据库访问：** sqlx（异步、连接池、编译期 SQL 校验）+ PostgreSQL 驱动

- **密码哈希：** Argon2（argon2 crate，加盐，替代 SHA-256）

- **大模型调用：** reqwest（OpenAI 兼容 HTTP 接口）

- **UUID生成：** Uuid

- **时间处理：** Chrono

- **数据导入导出：** 明文 JSON（导出为 `application/json` 文件 `taskflow_export.json`，导入解析同结构 JSON；不使用压缩）

#### 2.2.2 前端技术

- **框架：** React + TypeScript

- **构建工具：** Vite

- **桌面外壳：** Tauri v2（系统 WebView，产出原生安装包：macOS `.dmg`、Windows `.exe`(NSIS)）

- **HTTP 请求：** fetch（统一封装，携带 X-Session-Id 及可选 X-LLM-* 请求头）

- **图表库：** Chart.js（沿用 V1 图表逻辑）

- **路由：** react-router-dom（落地页/登录/注册/下载/应用等页面路由）

- **Markdown：** react-markdown（渲染 Agent 助理的流式答复）

- **状态管理：** Zustand（全局会话 / 任务 / 设置 / LLM 配置）+ React 内置状态

#### 2.2.3 数据存储

- **中央数据库：** Supabase 托管 PostgreSQL（唯一数据源，多用户、跨设备）

- **主要数据表：** users、sessions、tasks（含子任务）、task_templates、checkins

- **回收站：** 由 tasks 表的 deleted_at 软删除实现，不再单独建表

- **向量检索预留：** 后续可启用 pgvector 扩展为语义检索建向量索引

- **说明：** V1.0 的本地 JSON 文件存储方案已废弃

#### 2.2.4 大模型集成（V2.0 新增）

- **接口规范：** OpenAI 兼容 Chat Completions

- **默认模型：** deepseek-v4-flash（base_url / 模型名 / 密钥均可配置，可切换到 OpenAI 或其他兼容服务）

- **结构化输出：** 使用 function calling / response_format(JSON Schema) 约束解析、拆解、检索的结果格式

- **密钥管理：** 用户自带 API key（base_url / 模型名亦可自定义）。可仅在客户端本地保管、随请求头传递（后端用完即弃），也可保存到账户（持久化于 users 表）实现跨设备同步。读取优先级：请求头 > 账户设置 > 服务端默认（迁移 0002 为 users 表新增 llm_api_key / llm_model / llm_base_url 三列）

- **容错降级：** 调用超时或失败时降级到手动表单，核心任务管理不受影响

### 2.3 数据模型设计

数据库为 PostgreSQL（Supabase 托管）。以下为主要表的设计（V2.0 为全新建库，不迁移 V1.0 数据）。

#### 2.3.1 用户表 (users)

```sql
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone         VARCHAR(11) UNIQUE NOT NULL,        -- 登录账号(11位)
    password_hash TEXT NOT NULL,                      -- Argon2 加盐哈希
    summary_tone  VARCHAR(20) NOT NULL DEFAULT '温暖鼓励型', -- 推荐/总结语气偏好
    llm_api_key   TEXT NOT NULL DEFAULT '',            -- 账户级大模型设置(迁移 0002)，空=走服务端默认
    llm_model     TEXT NOT NULL DEFAULT '',            -- 同上，可跨设备同步
    llm_base_url  TEXT NOT NULL DEFAULT '',            -- 同上
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

> 说明：`llm_*` 三列由迁移 `0002_llm_settings.sql` 增加，用于把用户的大模型设置从浏览器 localStorage 升级为账户级持久化，实现换设备免重填。

#### 2.3.2 会话表 (sessions)

```sql
CREATE TABLE sessions (
    id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),  -- 即 session_id
    user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at TIMESTAMPTZ NOT NULL                          -- 会话持久化，重启不丢
);
```

#### 2.3.3 任务表 (tasks，含一级子任务)

```sql
CREATE TABLE tasks (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    parent_id    UUID REFERENCES tasks(id) ON DELETE CASCADE, -- NULL=顶层任务/任务组头；非空=子任务(仅一级)
    title        TEXT NOT NULL,
    description  TEXT NOT NULL DEFAULT '',
    completed    BOOLEAN NOT NULL DEFAULT false,
    category     VARCHAR(10) NOT NULL DEFAULT '其他',          -- 学习/工作/生活/家庭/其他
    star_rating  SMALLINT NOT NULL DEFAULT 0,                  -- 0-5 重要程度
    sort_order   INT NOT NULL DEFAULT 0,                       -- 同组内建议执行顺序
    start_date   TIMESTAMPTZ,
    deadline     TIMESTAMPTZ,
    deleted_at   TIMESTAMPTZ,                                  -- 非空=在回收站(软删除)
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at TIMESTAMPTZ
);
CREATE INDEX idx_tasks_user   ON tasks(user_id);
CREATE INDEX idx_tasks_parent ON tasks(parent_id);
```

说明：
- **任务组 / 子任务**：一个任务组 = 一个父任务（`parent_id` 为空）+ 若干子任务（`parent_id` 指向父）。仅允许一级——应用层保证子任务所指父任务自身 `parent_id` 必为空。父任务作为容器，进度由子任务完成比例计算。普通独立任务即 `parent_id` 为空且无子任务的行。
- **回收站**：用 `deleted_at` 软删除替代 V1.0 的独立回收站文件，`deleted_at` 非空即在回收站。

#### 2.3.4 任务模板表 (task_templates)

## 3. API文档

> **本章完成人：角色4（前端开发 + 文档编写），接口实现由角色1、角色2 提供**

### 3.1 基础信息

- **Base URL：** 部署环境为后端服务地址（生产经 Caddy 反代，前端以相对路径 `/api` 访问）；本地开发后端为 `http://localhost:8090/api`，前端 Vite（:5173）将 `/api` 代理到该地址；桌面版与网页版指向同一后端

- **请求格式：** JSON

- **响应格式：** JSON

- **认证方式：** Session ID（通过 `X-Session-Id` 请求头传递）；会话持久化于数据库，服务重启不失效；密码以 Argon2 校验

- **大模型请求头（`/ai/*` 与 `/ai/agent` 接口）：** `X-LLM-Key`、`X-LLM-Model`、`X-LLM-Base-Url` 均为可选。后端解析大模型配置的优先级为「请求头 > 账户设置（users 表）> 服务端默认（deepseek-v4-flash）」；请求头方式用完即弃，账户设置方式持久化于数据库以便跨设备同步

### 3.2 通用响应格式

{
    "success": true/false,
    "message": "提示信息",
    "data": {} // 可选，返回数据
}

### 3.11 智能助理 Agent API（V2.0 新增）

#### 3.11.1 Agent 对话（多轮工具调用 + 确认闸 + SSE 流式）

- **URL：** `POST /ai/agent`

- **请求头：** `X-Session-Id`（必填）、`X-LLM-*`（可选，见 3.1）

- **请求体：**

{
    "messages": [],                 // 前端持有的对话历史(不含 system)，原样回传；首轮为空数组
    "user_input": "这周学习类还剩哪些没做",  // 本轮用户新消息(首轮/追问时)
    "decision": null                // 可选：对上一条 pending 写操作的确认/拒绝
}

- **decision（确认/拒绝上一条拟改动）：** `{"tool_call_id": "call_xxx", "approved": true, "note": null}`

- **响应：** `text/event-stream`（SSE）。事件类型：

- `start`：模型开始新的一段输出

- `thinking`：推理过程增量（前端可折叠）

- `delta`：最终答复文本增量（Markdown）

- `tool`：某个工具已执行，携带 `{name, args, ok, result}`

- `done`：收尾，携带 `{messages, reply, pending}`；`messages` 为更新后的完整历史（前端存下用于下一轮回传），`pending` 非空表示有一条待确认的写操作

- `error`：出错（如大模型不可用）

- **可用工具：** 读工具 `list_tasks` / `get_stats`（即时执行）；写工具 `create_task` / `update_task` / `delete_task`（不直接落库，作为 `pending` 返回，等 `decision.approved=true` 后才真正写入）

- **说明：** 助理无状态——服务端不保存对话，历史由前端持有回传；写操作确认闸确保"用户确认后才生效"，与 1.4.2「杜绝静默写入」一致。若历史最后一条 assistant 带悬空的 tool_calls，后端会幂等补齐，保证多轮驱动稳健

#### 4.2.13 自动化单元测试覆盖（43 项概况）

| 模块 | 覆盖点 |
| --- | --- |
| util | 北京时区当日起点、`parse_flexible_date` 多格式/多时区解析 |
| tasks | 分类合法性、星级钳位 0-5、日期字段「缺省/null/有值」三态区分、`per_page` 下限钳位（防 `per_page=0` 除零） |
| auth | 手机号 11 位校验、密码 ≥6 位、Argon2 哈希往返且非明文 |
| ai | LLM 配置三级优先级、JSON 结果容错包裹（items/顶层数组/单对象）、`base_url` SSRF 形态校验（拒链路本地/元数据、放行本地自建 LLM）、无自带 key 时服务端 key 不外发 |
| checkin | 首签起步、连续 +1、断签重置、当日重复、多日间隔重置 |

#### 示例3：使用Claude生成Chart.js图表代码

提示词：

请帮我使用Chart.js生成一个饼图，显示任务的完成状态分布。
需要显示：已完成、待做、已过期三个状态的数量和百分比。

AI生成的代码：

function renderPieChart(stats) {
    const ctx = document.getElementById("pie-chart").getContext("2d");
    const labels = ["已完成", "待做中", "已过期"];
    const values = [stats.completed, stats.pending, stats.expired];
    const colors = ["#27ae60", "#667eea", "#e74c3c"];
    
    pieChart = new Chart(ctx, {
        type: "pie",
        data: {
            labels: labels,
            datasets: [{
                data: values,
                backgroundColor: colors,
                borderWidth: 2,
                borderColor: "#ffffff"
            }]
        },
        options: {
            responsive: true,
            plugins: {
                tooltip: {
                    callbacks: {
                        label: function(context) {
                            const total = context.dataset.data.reduce((a, b) => a + b, 0);
                            const percentage = ((context.parsed / total) * 100).toFixed(1);
                            return `${context.label}: ${context.parsed} (${percentage}%)`;
                        }
                    }
                }
            }
        }
    });
}

#### 示例4：使用 Claude 设计 Agent 工具 schema 与确认闸（V2.0）

提示词：

我在做一个任务管理助理（Agent 模式），要让大模型通过 function calling 读写任务，
但写操作（增删改）必须先让用户确认再落库。请帮我设计工具 schema 与"确认闸"的后端流程。

AI 给出的方案（节选）：

- 工具分读/写两类：list_tasks / get_stats 即时执行；create_task / update_task / delete_task
  不直接落库，而是作为 pending「拟改动」返回给前端
- 前端展示 pending 卡片，用户确认（approved=true）后，后端凭 tool_call_id 找回原始参数再真正写入
- 对话历史由前端持有并回传，服务端无状态；末条 assistant 若带悬空 tool_calls 则幂等补齐一条结果，保证多轮驱动稳健
- 思考过程、工具调用与最终答复经 SSE 流式返回，前端可折叠思考/工具轨迹、Markdown 渲染答复

（该方案对应当前 `src/agent.rs` 的实现。）
